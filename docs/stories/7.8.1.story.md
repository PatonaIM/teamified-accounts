# Story 7.8.1: Employee Selection UI for Payroll Processing

## Status
‚úÖ Ready for Review - October 4, 2025

## Story
**As a** payroll administrator,
**I want** a user-friendly interface to select specific employees for payroll processing,
**so that** I can exclude employees who shouldn't be paid in a particular payroll run (e.g., employees on unpaid leave, terminated employees with pending clearance, or partial payroll runs for testing).

## Acceptance Criteria
1. **Employee Selection Dialog:** Modal dialog showing all active employees for the selected country
2. **Multi-Select Interface:** Checkboxes for selecting/deselecting individual employees
3. **Bulk Actions:** "Select All" and "Deselect All" buttons for convenience
4. **Search & Filter:** Search by name/ID, filter by department/location
5. **Employee Count Indicator:** Display count of selected vs. total employees (e.g., "15 of 100 selected")
6. **Visual Feedback:** Clear indication when custom selection is active vs. "all employees"
7. **Clear Selection:** Ability to reset to default "all employees" mode
8. **Integration:** Seamlessly integrate with existing ProcessingControlPanel component
9. **Validation:** Prevent processing with zero employees selected
10. **Performance:** Handle 1000+ employee lists efficiently with virtualization

## Dependencies
- **Story 7.8:** Advanced Payroll Administration & Monitoring ‚úÖ **COMPLETE** (provides backend support)
  - **CRITICAL:** Backend `StartPayrollProcessingDto` already accepts `userIds` parameter
  - **CRITICAL:** Backend `PayrollProcessingService` already handles specific employee processing
  - This story ONLY adds the frontend UI to populate the existing backend parameter

## Dev Notes

### Integration with Story 7.8

This story is a **pure frontend enhancement** to Story 7.8. No backend changes are required.

**Existing Backend APIs (Story 7.8):**
- ‚úÖ `POST /api/v1/payroll/admin/processing/start` - Already accepts `userIds` parameter
- ‚úÖ `PayrollProcessingService.startPayrollProcessing()` - Already handles `dto.userIds`
- ‚úÖ `PayrollProcessingService.getSpecificEmployees()` - Already implemented
- ‚ö†Ô∏è May need: `GET /api/v1/payroll/admin/employees?country=IN` - Check if exists, create if not

**Frontend Architecture:**
- Dialog component is standalone and reusable
- State management via React useState in ProcessingControlPanel
- No global state needed (selection is per-session, not persisted)
- Dialog closes on apply/cancel, selection persists until "Clear Selection" or page refresh

### UI/UX Design Patterns

**Visual Hierarchy:**
1. Primary action: "Start Processing" (always visible, most prominent)
2. Secondary action: "Select Employees" (above Start Processing, less prominent)
3. Selection indicator: Chip/Badge (only when custom selection active)
4. Clear action: IconButton (small X, only when custom selection active)

**User Flow:**
1. User lands on Processing Control tab ‚Üí sees "All employees will be processed"
2. User clicks "Select Employees" ‚Üí dialog opens with all employees
3. User searches/filters/selects employees ‚Üí sees count update "15 of 100"
4. User clicks "Apply Selection" ‚Üí dialog closes, badge shows "15 Employees Selected"
5. User clicks "Start Processing" ‚Üí confirmation shows "Process 15 selected employees?"
6. User confirms ‚Üí API call includes `userIds: [...]`

**Default Behavior (No Changes to Existing):**
- If user never clicks "Select Employees", system processes ALL employees (current behavior)
- If user opens dialog but clicks "Cancel", selection is discarded
- If user clicks "Clear Selection" after applying, reverts to ALL employees

### Material-UI Components Used

- **Dialog:** `maxWidth="md"`, `fullWidth`
- **DataGrid:** `checkboxSelection`, `disableRowSelectionOnClick`, `rowHeight={52}`
- **TextField:** Search input with `InputAdornment` (Search icon)
- **Autocomplete:** Multi-select for department/location filters
- **Button:** "Select All", "Deselect All", "Cancel", "Apply Selection"
- **Chip:** Selection count indicator with `onDelete` for clear
- **IconButton:** Clear selection (X icon)
- **Alert:** Warning when zero employees selected

### Performance Considerations

**Large Employee Lists (1000+ employees):**
- Use DataGrid's built-in virtualization (only renders visible rows)
- Implement client-side search/filter (avoids API calls on every keystroke)
- Load employee list once on dialog open, cache in component state
- Use `useMemo` for filtered/searched employee list
- Debounce search input (300ms) to reduce re-renders

**API Response Size:**
- Employee list API should return minimal fields (id, name, employeeId, department, location)
- Avoid loading full employee records with all fields
- Response size for 1000 employees: ~200KB (acceptable)

### Business Rules

1. **Employee Eligibility:**
   - Only employees with **active employment records** in the selected country are shown
   - Inactive/terminated employees are excluded from the list
   - Employee must have employment record for the payroll period dates

2. **Selection Persistence:**
   - Selection persists within the current session (until page refresh or logout)
   - Selection is cleared when:
     - User clicks "Clear Selection"
     - User changes country (selection is country-specific)
     - User changes period (selection is period-specific)
     - Page is refreshed

3. **Validation Rules:**
   - Minimum 1 employee must be selected if custom selection is active
   - Cannot start processing with zero employees
   - Show warning Alert if validation fails

4. **Use Cases:**
   - **Partial Payroll Run:** Process only specific employees (e.g., mid-month joiners)
   - **Test Payroll:** Process 1-2 employees to verify calculations before full run
   - **Exclude Employees:** Deselect employees who shouldn't be paid (unpaid leave, pending clearance)
   - **Department-Specific:** Select all employees in a department for separate processing
